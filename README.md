# AgriPay

> A non-KYC community marketplace for Kenyan agricultural cooperatives — global Bitcoin payments via Lightning, local cash-out via M-Pesa, identity via Nostr.

AgriPay connects global buyers to Kenyan farmers through a Bitcoin-native payments layer. It runs as a **Fedi mini app** — any member of a Fedi federation authenticates automatically using their Nostr identity, with zero accounts, zero passwords, and zero KYC. Buyers pay using **Lightning Network** invoices generated by **BTCPay Server**. Farmers accumulate a sats balance and cash out to **M-Pesa** on demand. The platform handles FX conversion, balance accounting, event sourcing, and background reconciliation — all in a single Rust/Axum service backed by PostgreSQL.

## Features

- **Fedi mini app** — loads inside Fedi's in-app browser; members sign in automatically via their Nostr key (NIP-07 + NIP-98), no account creation required
- **Non-KYC, open access** — identity is a Nostr public key; new farmers are auto-registered on first sign-in
- **BTCPay Server integration** — invoice creation and HMAC-SHA256 verified webhooks; no self-hosted Lightning node required
- **Farmer balance ledger** — internal accounting layer decouples payment receipt from cash-out; farmers accumulate sats over multiple payments
- **M-Pesa cash-out on demand** — farmers explicitly trigger B2C transfers; atomic balance locking prevents double-withdrawal
- **Marketplace orders** — farmers list crop sell-orders; operators fill them
- **Multi-state payment machine** — `created → invoice_created → bitcoin_received → credited_to_farmer → failed`
- **Withdrawal machine** — `pending → processing → disbursing_mpesa → completed | failed`
- **Event sourcing** — every payment state transition recorded in `payment_events` with full JSONB payload
- **Live FX oracle** — CoinGecko BTC/KES + BTC/USD with configurable cache TTL and stale-rate rejection
- **Background workers** — invoice expiry, cash-out queue processor, and stuck-disbursement reconciliation with automatic refund
- **Role-based access** — `admin`, `operator`, and `farmer` roles; Nostr login issues farmer-scoped JWTs
- **Structured JSON logging** — toggleable between human-readable and production JSON via `LOG_FORMAT`
- **PostgreSQL** — full ACID transactions, JSONB event data, CHECK constraints on all status columns

## Authentication

AgriPay uses **Nostr NIP-98 HTTP Auth** — no passwords, no accounts.

1. Fedi injects `window.nostr` (NIP-07) into the mini app webview
2. The frontend creates a signed kind-27235 event and POSTs it to `/api/auth/nostr`
3. The backend verifies the BIP-340 Schnorr signature and issues a JWT
4. On first visit, a farmer record is auto-created for the Nostr pubkey — zero friction

Outside Fedi, the app shows a "Get Fedi" prompt instead.

## Payment Flows

### Flow A — Global Bitcoin Payment

```
Buyer → BTCPay Invoice → Lightning payment received
  → BTCPay webhook (InvoiceSettled)
  → payment: bitcoin_received
  → farmer balance: available_sats += amount   ← atomic DB transaction
  → payment: credited_to_farmer
```

### Flow B — Farmer Cash-Out (M-Pesa)

```
Farmer → POST /api/wallet/withdraw (amount_kes)
  → rate fetched, KES → sats conversion
  → balances: available_sats -= X, locked_sats += X  ← FOR UPDATE lock
  → withdrawal created (pending)
  → cash-out worker picks up (every 30 s)
  → M-Pesa B2C initiated → disbursing_mpesa
  → Safaricom callback → completed (locked_sats -= X)
                       → failed    (locked_sats -= X, available_sats += X)
```

### Flow C — Stuck Disbursement Recovery

```
Reconciliation worker (every 5 min)
  → finds disbursing_mpesa withdrawals older than DISBURSE_STALE_SECS
  → auto-fails + refunds locked → available  ← atomic, idempotent
  → logs critical alert for manual review
```

## Quick Start

### 1. Clone

```bash
git clone https://github.com/comwanga/Agri-pay.git
cd Agri-pay
```

### 2. Configure environment

```bash
cp .env.example .env
```

At minimum, update these in `.env`:

```env
DATABASE_URL=postgresql://agripay:agripay_dev@localhost:5433/agri_pay?sslmode=disable
JWT_SECRET=<generate with: openssl rand -base64 48>
BTCPAY_URL=https://your-btcpay-instance
BTCPAY_API_KEY=your_api_key
BTCPAY_STORE_ID=your_store_id
BTCPAY_WEBHOOK_SECRET=<generate with: openssl rand -hex 32>
```

### 3. Start PostgreSQL

```bash
docker compose up postgres -d
```

Wait ~5 seconds for the health check to pass. Migrations run automatically on first backend start.

### 4. Run the backend

```bash
cargo run
```

Server starts on `http://0.0.0.0:3001`. Migrations are applied automatically on first run.

### 5. Run the frontend (dev)

```bash
cd frontend
npm install
npm run dev
```

Frontend dev server on `http://localhost:5173`. For Nostr auth to work, open the deployed GitHub Pages URL inside Fedi and point `VITE_API_URL` to a publicly reachable backend.

## Frontend Deployment (GitHub Pages)

The frontend is deployed automatically to **[comwanga.github.io/Agri-pay](https://comwanga.github.io/Agri-pay/)** on every push to `main` via GitHub Actions.

To connect it to a backend:

1. Deploy the Rust backend to a public server (VPS, Railway, Fly.io, etc.)
2. In the GitHub repo: **Settings → Variables → Actions → New variable**
   - Name: `VITE_API_URL`
   - Value: `https://your-backend.example.com/api`
3. Re-run the `Deploy frontend to GitHub Pages` workflow

For testing with a local backend, use [ngrok](https://ngrok.com):

```bash
ngrok http 3001
# Set VITE_API_URL to the ngrok HTTPS URL + /api, then redeploy
```

## API Reference

All endpoints are prefixed with `/api`. Protected routes require `Authorization: Bearer <token>`.

### Auth

| Method | Path | Role | Description |
|---|---|---|---|
| `POST` | `/auth/nostr` | — | Nostr NIP-98 login. Accepts a signed kind-27235 event `{event: {...}}`. Auto-registers farmers by pubkey. Returns JWT. |
| `POST` | `/auth/login` | — | Password login `{username, password}`. Supports admin and operator accounts. |
| `POST` | `/auth/register` | admin | Create a user account `{username, password, role, farmer_id?}` |

### Farmers

| Method | Path | Role | Description |
|---|---|---|---|
| `GET` | `/farmers` | admin, operator | List all farmers |
| `POST` | `/farmers` | admin | Create farmer `{name, phone, cooperative, pin?}` |
| `GET` | `/farmers/:id` | admin, operator, self | Get a single farmer |
| `PUT` | `/farmers/:id` | admin | Update farmer fields |
| `DELETE` | `/farmers/:id` | admin | Delete farmer |

### Payments

| Method | Path | Role | Description |
|---|---|---|---|
| `GET` | `/payments?page=1&per_page=50&farmer_id=&status=` | admin, operator | List payments with optional filters |
| `POST` | `/payments` | admin, operator | Create payment + BTCPay invoice `{farmer_id, amount_kes, crop_type?, notes?}` |
| `GET` | `/payments/:id` | admin, operator, self | Get payment (farmers see only their own) |

### Wallet

| Method | Path | Role | Description |
|---|---|---|---|
| `GET` | `/wallet/balance` | farmer | Get available and locked sats |
| `POST` | `/wallet/withdraw` | farmer | Request M-Pesa cash-out `{amount_kes}` |

### Orders

| Method | Path | Role | Description |
|---|---|---|---|
| `GET` | `/orders` | any | List open crop sell-orders |
| `POST` | `/orders` | farmer | Create order `{crop_type, quantity_kg, price_per_kg_sats}` |
| `PUT` | `/orders/:id/fill` | admin, operator | Mark order as filled |
| `DELETE` | `/orders/:id` | farmer (own), admin | Cancel order |

### Oracle

| Method | Path | Role | Description |
|---|---|---|---|
| `GET` | `/oracle/rate` | any | Current BTC/KES and BTC/USD rate with cache metadata |

### Webhooks

| Method | Path | Auth | Description |
|---|---|---|---|
| `POST` | `/webhooks/btcpay` | HMAC-SHA256 `BTCPay-Sig` header | BTCPay `InvoiceSettled` event — credits farmer balance |
| `POST` | `/webhooks/mpesa/:secret/result` | Secret in path | M-Pesa B2C result callback |
| `POST` | `/webhooks/mpesa/:secret/timeout` | Secret in path | M-Pesa B2C timeout callback — refunds locked sats |

### Health

| Method | Path | Description |
|---|---|---|
| `GET` | `/health` | Returns `{"status":"ok","version":"0.2.0"}` |

## Payment State Machine

### Payments

```
created
  └─▶ invoice_created   (BTCPay invoice generated)
        └─▶ bitcoin_received   (InvoiceSettled webhook)
              └─▶ credited_to_farmer   (balance updated atomically)
  └─▶ failed   (BTCPay error or invoice expiry)
```

### Withdrawals

```
pending
  └─▶ processing       (cash-out worker acquired the row)
        └─▶ disbursing_mpesa   (B2C request accepted by Safaricom)
              ├─▶ completed    (result callback: code 0)
              └─▶ failed       (result/timeout callback, or reconciliation auto-fail)
                               → locked_sats refunded to available_sats
```

## Database Schema

| Table | Purpose |
|---|---|
| `users` | Admin and operator accounts with bcrypt password hashes |
| `farmers` | Cooperative member registry; `nostr_pubkey` links Fedi identities |
| `balances` | One row per farmer — `available_sats` and `locked_sats` |
| `payments` | BTCPay invoice lifecycle records with FX snapshot |
| `payment_events` | Append-only audit trail (JSONB) for every state transition |
| `withdrawals` | M-Pesa cash-out requests with full status lifecycle |
| `orders` | Farmer crop sell-orders for the marketplace |
| `rate_cache` | BTC/KES and BTC/USD rate history from CoinGecko |
